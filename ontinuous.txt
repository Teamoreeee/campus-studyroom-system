     - ❌ 一键清除功能 —— 尚未实现

     由于上一次对话结束时触发 API token 上限，最后一批前端修改未做最终编译验证，
     因此本计划会先完成该验证，再实现"一键清除"，最后做全量测试。

     需求澄清结果

     - 清除对象："我的预约"页面中已过期 / 已取消的预约记录
     - 删除方式：物理删除（从数据库真正删除）

     推荐方案

     在 ReservationFrontendController
     增加一个后端接口，物理删除当前用户的以下预约：
     1. status = 'cancelled' 或 status = 'timeout' 的记录；
     2. 或 status 仍为 pending/confirmed、但 end_time < 当前时间
     的记录（即前端显示为"已超时"但尚未被后端标记的记录）。

     前端在"我的预约"页面增加"清空已过期/已取消"按钮，点击后二次确认，成功后刷新
     列表。

     待修改文件

     后端

     1. backend/campus-reservation/src/main/java/com/campus/reservation/service/
     ReservationService.java
       - 新增方法签名：int clearUserExpiredReservations(Long userId);
     2. backend/campus-reservation/src/main/java/com/campus/reservation/service/
     impl/ReservationServiceImpl.java
       - 实现物理删除逻辑，使用 baseMapper.delete(wrapper) 绕过 @TableLogic
     逻辑删除
       - 条件：user_id = ? AND (status IN ('cancelled','timeout') OR (status IN 
     ('pending','confirmed') AND end_time < NOW()))
     3. backend/campus-reservation/src/main/java/com/campus/reservation/controll
     er/ReservationFrontendController.java
       - 新增 DELETE /api/reservation/reservations/clear-expired
       - 从 JWT 提取 userId，调用 service，返回删除条数

     前端

     4. frontend/src/api/reservation.ts
       - 新增 clearExpiredReservations: () => 
     request.delete('/reservation/reservations/clear-expired')
     5. frontend/src/views/reservations/ReservationList.vue
       - 在筛选区增加 "清空已过期/已取消" 按钮
       - 增加确认对话框
       - 成功后 ElMessage.success 并刷新列表

     实施步骤

     1.
     检查当前服务状态，确认最后一批前端修改已生效；如未生效则重新编译/重启前端。
     2. 修改 ReservationService 接口与实现，实现物理删除逻辑。
     3. 修改 ReservationFrontendController，暴露新的 DELETE 接口。
     4. 修改前端 API 封装与 ReservationList.vue 页面。
     5. 后端 mvn clean package -DskipTests -pl backend/campus-reservation -am
     编译打包。
     6. 重启 reservation 服务。
     7. 验证：
       - 未登录调用接口返回 401；
       - 登录用户只能删除自己的过期/已取消记录；
       - 前端点击按钮后列表刷新，记录消失；
       - 数据库中对应行被物理删除。

     验证方式

     - 浏览器访问 http://localhost:3000，使用预填账号登录
     - 在"我的预约"页面制造/确认存在已取消或已超时记录
     - 点击"清空已过期/已取消"，确认二次弹窗
     - 观察列表刷新后记录消失，同时用 MySQL 查询 reservation 表确认行被物理删除

     风险与注意事项

     - 物理删除不可恢复，已在二次确认文案中强调。
     - @TableLogic 默认会逻辑删除，因此必须使用 baseMapper.delete(wrapper)
     执行真实 DELETE。
     - 仅删除当前用户记录，通过 JWT 提取 userId 并在 wrapper 中过滤，避免越权。
     - 建议同步检查 application.yml 中是否仍残留之前硬编码的智谱 API

● 已修复 ✅

  原因

  之前网关的 JWT 过滤器只检查 token 有没有、格式对不对，没有验证是否过期。过期
  token 被直接转发给后端服务，后端 JwtUtils 解析时抛异常，又被
  GlobalExceptionHandler 捕获成 500 服务器内部错误。

  前端看到 500 就不会触发"token 过期 →
  刷新/跳转登录"的逻辑，所以显示了那个报错。

  修复

  在网关统一做 JWT 过期校验：

  1. backend/campus-gateway/pom.xml — 添加 jjwt 依赖
  2. backend/campus-gateway/.../GatewayConfig.java — JWT 过滤器现在会真正解析
  token，过期或无效都直接返回 401
  3. 响应体使用 UTF-8 编码，中文消息正常显示：{"code":401,"message":"token 
  无效或已过期"}

  前端 request.ts 本身已经有 401 处理逻辑：
  - 先尝试用 refreshToken 刷新
  - 刷新失败就清除登录状态，自动跳回 /login

  现在整条链路通了。
