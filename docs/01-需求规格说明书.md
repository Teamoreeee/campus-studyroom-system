## 1 引言

### 1.1 编写目的

本需求规格说明书旨在全面、准确地描述校园自习室预约系统的功能需求、非功能需求及约束条件，作为系统架构设计、编码实现、测试验收的依据。本文档面向项目团队成员、指导教师及评审人员。

### 1.2 项目背景

当前高校自习室资源管理面临诸多挑战：一是自习室座位资源紧张，高峰期一座难求，学生经常需要逐层逐间寻找空座，浪费时间且体验差；二是"占座不学"现象普遍，预约后不签到或签到后短暂离开，造成资源浪费；三是考勤管理依赖人工，效率低下且易出错；四是缺乏数据驱动的智能推荐，学生难以快速找到适合自己学习习惯的自习室。

本系统通过构建微服务架构的校园自习室预约平台，结合AI智能推荐与异常分析能力，实现自习室资源的数字化、智能化管理，提升资源利用率和学生学习体验。

## 2 系统概述

### 2.1 系统目标

构建一个基于Spring Cloud Alibaba微服务架构的校园自习室预约系统，实现以下核心目标：

1. **资源数字化**：实现自习室、座位、时间段的全数字化管理
2. **预约智能化**：通过AI算法为学生推荐最适合的自习室和座位
3. **考勤自动化**：自动签到/签退，AI智能分析考勤异常
4. **管理精细化**：违规行为智能检测，资源利用数据驱动
5. **架构云原生化**：基于Docker + Kubernetes的云原生部署，全链路可观测

### 2.2 用户特点

| 用户角色 | 特点描述 | 使用频率 |
|----------|----------|----------|
| 学生 | 18-25岁在校大学生，移动端使用为主，期望快速预约、智能推荐 | 高频（每日1-3次） |
| 管理员 | 教务人员或自习室管理人员，需高效管理资源、处理异常 | 中频（每日1次） |
| 超级管理员 | 信息中心技术人员，负责系统运维与配置 | 低频（每周1-2次） |

## 3 功能需求

### 3.1 用户管理模块

#### 3.1.1 功能概述

用户管理模块负责系统用户的注册、认证、授权及个人信息管理，采用RBAC权限模型，支持学生、管理员、超级管理员三种角色。

#### 3.1.2 功能清单

| 编号 | 功能名称 | 优先级 | 描述 |
|------|----------|--------|------|
| UM-01 | 用户注册 | 高 | 学生通过学号注册，验证学号有效性，设置密码 |
| UM-02 | 用户登录 | 高 | 支持账号密码登录，JWT令牌认证 |
| UM-03 | Token刷新 | 高 | JWT Token自动刷新机制，无感续期 |
| UM-04 | 个人信息管理 | 中 | 修改头像、昵称、联系方式、学习偏好 |
| UM-05 | 密码管理 | 中 | 修改密码、找回密码（验证邮箱） |
| UM-06 | 角色权限管理 | 高 | 管理员分配角色，基于RBAC的权限控制 |
| UM-07 | 用户状态管理 | 中 | 管理员启用/禁用用户账号 |
| UM-08 | 学习偏好设置 | 中 | 设置偏好的自习室类型、时间段、楼层等，用于AI推荐 |

#### 3.1.3 业务规则

- 学号唯一，注册时校验学号是否已存在
- 密码需满足8位以上，包含字母和数字
- JWT Token有效期2小时，Refresh Token有效期7天
- 用户被禁用后无法登录，已有Token立即失效

### 3.2 自习室管理模块

#### 3.2.1 功能概述

自习室管理模块负责自习室信息、座位、设施和时间段的管理，为预约模块提供基础数据支撑。

#### 3.2.2 功能清单

|------|----------|--------|------|

| RM-02 | 自习室状态管理 | 高 | 开放/关闭/维护状态切换 |
| RM-03 | 座位管理 | 高 | 为自习室添加/删除/修改座位，设置座位属性（靠窗/靠墙/有电源/普通） |
| RM-04 | 座位状态管理 | 高 | 座位实时状态（空闲/已预约/使用中/维护中） |
| RM-05 | 时间段模板 | 中 | 配置自习室开放时间段，支持自定义时段 |
| RM-06 | 设施标签 | 中 | 为自习室添加设施标签（WiFi、空调、电源、白板等） |
| RM-07 | 自习室搜索 | 高 | 按教学楼、楼层、设施、可用状态搜索自习室 |
| RM-08 | 实时座位图 | 高 | 可视化展示自习室座位布局与实时占用状态 |

#### 3.2.3 业务规则

- 每个自习室必须有唯一的名称和位置标识
- 座位编号在同一自习室内唯一
- 自习室关闭后，已有预约自动取消并通知用户
- 座位状态实时更新，通过Redis缓存保证一致性

### 3.3 预约管理模块

#### 3.3.1 功能概述

预约管理模块是系统核心业务模块，负责学生预约座位、取消预约、预约冲突检测和并发控制。

#### 3.3.2 功能清单

| 编号 | 功能名称 | 优先级 | 描述 |
|------|----------|--------|------|
| RV-01 | 创建预约 | 高 | 学生选择自习室、座位、时间段提交预约 |
| RV-02 | 取消预约 | 高 | 学生在规定时间内取消已有预约 |
| RV-03 | 预约冲突检测 | 高 | 实时检测座位时间段是否冲突，防止重复预约 |
| RV-04 | 预约超时处理 | 高 | 预约开始后15分钟未签到，自动取消并释放座位 |
| RV-05 | 预约限制 | 高 | 同一学生同一时间段只能预约一个座位 |
| RV-06 | 每日预约上限 | 中 | 限制学生每日预约次数（最多3次），防止恶意占用 |
| RV-07 | 预约历史 | 中 | 查看个人预约历史记录 |
| RV-08 | 预约统计 | 中 | 管理员查看预约数据统计、趋势分析 |

#### 3.3.3 业务规则

- 预约时间段不可重叠（同一座位同一时间只能被一人预约）
- 使用Redis分布式锁保证高并发下预约不超卖
- 预约开始前30分钟可取消，开始后不可取消（需签退）
- 超时未签到自动取消预约，同时记录一次违规
- 处于处罚期内的用户无法创建预约

### 3.4 考勤管理模块

#### 3.4.1 功能概述

考勤管理模块负责学生签到/签退的记录与管理，与预约模块联动，为违规检测和AI异常分析提供数据基础。

#### 3.4.2 功能清单

| 编号 | 功能名称 | 优先级 | 描述 |
|------|----------|--------|------|
| AT-01 | 签到 | 高 | 学生在预约时间段内签到，验证身份与位置 |
| AT-02 | 签退 | 高 | 学生离开时签退，记录实际学习时长 |
| AT-03 | 自动超时签退 | 中 | 预约时间结束自动签退 |
| AT-04 | 考勤记录查询 | 中 | 查看个人考勤记录、学习时长统计 |
| AT-05 | 考勤统计 | 中 | 管理员查看考勤汇总、出勤率、平均时长 |
| AT-06 | 考勤异常标记 | 高 | 系统自动标记异常考勤（超时签到、短时签退等） |

#### 3.4.3 业务规则

- 签到必须在预约时间段内，且在预约开始后15分钟内
- 签退后自动释放座位，更新为空闲状态
- 签到后30分钟内签退视为"短时占座"，标记异常
- 考勤数据通过RabbitMQ异步写入，降低主流程延迟

#### 3.5.1 功能概述

违规管理模块负责记录和处罚违规行为，包括预约未签到、短时占座、频繁取消等，维护公平的资源使用秩序。

- 处罚期内用户无法创建新预约
- 申诉通过后立即解除处罚
- 违规记录保留一学期，学期末自动清除

### 3.6 AI智能模块

#### 3.6.1 功能概述

AI智能模块是系统的核心创新点，提供自习室智能推荐、考勤异常智能分析和RAG智能客服三大AI赋能功能。

#### 3.6.2 自习室智能推荐

| 编号 | 功能名称 | 优先级 | 描述 |
|------|----------|--------|------|
| AI-01 | 个性化推荐 | 高 | 基于用户历史行为和偏好，推荐最合适的自习室 |
| AI-02 | 实时可用性过滤 | 高 | 推荐结果仅包含当前有空座的自习室 |
| AI-03 | 推荐解释 | 中 | 大模型生成推荐理由的自然语言描述 |
| AI-04 | 相似用户推荐 | 中 | 基于协同过滤，推荐与相似用户群体偏好一致的自习室 |
| AI-05 | 内容匹配推荐 | 中 | 基于用户偏好特征（楼层、设施、时间段）匹配自习室 |
| AI-06 | 推荐反馈 | 低 | 用户对推荐结果进行评分，优化推荐算法 |

**推荐算法方案**：User-based协同过滤 + 内容过滤混合推荐
- 协同过滤层：计算用户间余弦相似度，找到Top-K相似用户，聚合相似用户偏好生成候选集
- 内容过滤层：提取用户偏好特征向量，与自习室属性向量计算相似度
- 融合排序：加权融合两路推荐分数（CF权重0.6 + 内容过滤权重0.4），结合实时可用性过滤
- 大模型增强：调用通义千问API对Top-N推荐结果生成自然语言推荐理由

#### 3.6.3 考勤异常智能分析

| 编号 | 功能名称 | 优先级 | 描述 |
|------|----------|--------|------|
| AI-07 | 规则引擎检测 | 高 | 基于预设规则检测明确异常（连续未签到、短时占座等） |
| AI-08 | 统计异常检测 | 中 | 统计分析用户考勤模式，检测隐性异常 |
| AI-09 | 异常风险评估 | 中 | 为每个用户生成考勤风险评分（0-100） |
| AI-10 | 异常预警 | 中 | 高风险用户自动预警通知管理员 |
| AI-11 | 异常报告生成 | 低 | 大模型生成异常分析报告 |

**异常检测方案**：规则引擎 + 统计分析双层架构
- 规则引擎层：使用Drools或自研规则引擎，配置异常检测规则，实时触发
- 统计分析层：统计用户历史考勤模式（平均学习时长、签到时间分布、自习室偏好变化），计算偏离度
- 风险评分：综合规则命中次数和统计偏离度，生成0-100风险评分

#### 3.6.4 RAG智能客服

| 编号 | 功能名称 | 优先级 | 描述 |
|------|----------|--------|------|
| AI-12 | 智能问答 | 高 | 用户提问关于自习室规则、预约流程等问题 |
| AI-13 | 知识库管理 | 中 | 管理员维护FAQ知识库内容 |
| AI-14 | 上下文对话 | 中 | 支持多轮对话，保持上下文 |
| AI-15 | 意图识别 | 中 | 识别用户意图（查询规则、投诉建议、预约帮助等） |

**RAG方案**：
- 知识库：构建自习室规则、预约流程、常见问题FAQ文档集
- 向量化：使用文本嵌入模型将知识库文档转为向量，存入向量存储
- 检索：用户提问 → 向量化 → 检索Top-K相关文档片段
- 生成：拼接检索结果 + 用户问题 → 调用通义千问API生成回答
- 目标准确率≥80%

## 6 约束与假设

### 6.1 约束条件

1. **技术栈约束**：后端必须使用Java 17 + Spring Boot 3.2.5 + Spring Cloud Alibaba 2023.0.1.2
2. **前端约束**：必须使用Vue 3 + TypeScript，禁止纯JavaScript
3. **数据库约束**：必须兼容MySQL 8.0和达梦8.0双数据库
4. **部署约束**：必须支持Docker + Kubernetes云原生部署
5. **AI约束**：AI功能必须真实解决业务痛点，禁止简单堆砌
6. **架构约束**：必须使用4+1视图方法进行架构建模
7. **文档约束**：架构设计文档≥38页，综合设计报告≥100页

### 6.2 假设条件

1. 校园网络基础设施完备，支持系统部署运行
2. Docker Desktop已安装，内置Kubernetes可用
3. 通义千问/文心一言API可正常访问
4. 达梦8数据库评估版可获取并安装
5. 学校提供学生学号验证接口（模拟实现）
6. 用户使用现代浏览器访问系统，兼容性良好

### 6.3 系统边界

- **包含**：预约管理、考勤管理、违规管理、AI推荐、AI异常分析、RAG客服、通知管理
- **不包含**：支付功能、社交功能、课程表集成、门禁硬件对接
